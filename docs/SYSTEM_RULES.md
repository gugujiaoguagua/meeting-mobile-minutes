# 会议闭环系统业务规则

本文档描述正式版需要保持一致的核心业务规则。

## 1. 会议召开判定

会议是否召开，不以人工勾选为准，而以上传会议录音、转写文稿或会议文档为准。

- 已上传文稿：视为会议已召开。
- 未上传文稿：视为会议未沉淀，会议列表中应提示未上传。
- 上传文件格式：第一阶段建议支持 Word、TXT、PDF。
- 录音识别可以后置，先支持已转写文本文件。

## 2. 会议纪要生成

上传会议文稿后，由 AI 生成统一会议闭环模板。

生成内容包括：

- 会议纪要摘要。
- 会议形成的决策。
- 会议待办事项。

当前产品方向已经取消独立的风险提示和独立的公司支持列表。需要公司支持的内容应挂在具体待办上。

## 3. 待办签批规则

主管生成并修正待办后，不能直接进入正式台账。

待办状态流转：

1. 主管生成待办。
2. 主管修正推进人、复核人、时间、目标、公司支持事项。
3. 提交管理者签批。
4. 管理者逐条签批。
5. 签批通过后进入正式待办台账。
6. 签批驳回后回到提交部门修改。

正式版应支持逐条签批，而不是整场会议一次性通过。

## 4. 待办完成规则

待办不能由推进人单方面标记完成。

正式完成流程：

1. 推进人执行任务。
2. 推进人提交完成。
3. 任务进入复核人的“待我复核”。
4. 复核人确认后，任务才算已完成。
5. 复核人驳回后，任务回到推进人继续修改。

默认复核人可以是会议主持人，但主管在创建待办时可以调整。

## 5. 公司支持事项

如果某个待办需要公司支持，应填写在该待办的“需要公司支持”字段中。

管理驾驶舱应自动汇总所有未完成的公司支持事项。

- 支持状态为待支持：显示在管理驾驶舱。
- 支持状态为已完成：从待支持列表中消失。
- 正式版需要记录谁完成、什么时候完成、支持说明。

## 6. 消息通知规则

消息通知是闭环系统的重要组成部分。

以下事件应产生消息：

- 主管提交待办签批。
- 管理者签批通过。
- 管理者驳回修改。
- 推进人提交完成。
- 复核人复核通过。
- 复核人驳回复核。
- 公司支持事项已完成。
- OKR KR 进入复核。
- OKR KR 复核通过。

第一阶段先做站内消息，第二阶段接企业微信机器人。

## 7. OKR 规则

OKR 项目包含：

- O：项目总目标，必须可量化、可测量。
- KR：关键结果，必须有衡量标准。
- PDCA：围绕 KR 拆解的执行动作。

PDCA 任务与普通待办一样，也应该进入员工“我的待办”。

KR 完成规则：

1. KR 下的 Plan、Do、Check、Act 动作全部完成。
2. 系统自动生成 KR 复核任务。
3. KR 复核任务进入 OKR 负责人的“我的待办 / 待我复核”。
4. OKR 负责人复核通过后，KR 才算正式完成并纳入项目统计。

## 8. 权限规则

第一阶段建议至少支持四类权限：

- 管理者：看管理驾驶舱、签批待办、查看全部任务和异常。
- 部门主管：新建会议、修正 AI 待办、提交签批、查看本部门看板。
- 员工：查看自己的待办、提交完成、查看与自己有关的会议纪要。
- OKR 负责人：查看负责的 OKR 项目、复核 KR 完成情况。

正式版应把权限做成配置，而不是写死角色。

## 9. 数据统计规则

所有看板统计必须从正式台账计算。

不能由前端页面自己拼数据，否则后续会出现统计口径不一致。

核心统计包括：

- 会议数量。
- 会议时长。
- 会议人工时。
- 形成待办数。
- 已完成数。
- 已提交待复核数。
- 逾期数。
- 公司支持事项数。
- 部门维度统计。
- OKR 项目维度统计。
